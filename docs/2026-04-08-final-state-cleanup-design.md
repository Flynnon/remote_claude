# 面向最终态的全量扫描与收敛清理设计

日期：2026-04-08

## 背景

当前仓库处于命令面、远程连接能力、Docker 回归链路持续调整阶段。分支内已经存在对 `README.md`、`bin/remote-claude`、`docs/docker-test.md` 和 Docker 测试脚本的修改，同时大量历史 `docs/superpowers/plans/`、`docs/superpowers/specs/` 内容已被删除。近期提交也集中在 CLI 命令面统一、remote 能力恢复、Docker 测试回归等方向。

在这种状态下，直接做“全量扫描并删除所有看起来无用的功能”风险较高，容易误删当前分支仍然需要保留的主路径能力。因此本设计选择一种面向最终态但保守收敛的方案：只收敛到已经能被代码、文档、测试、近期提交共同证明的主路径能力，并删除高置信度异常残留。

## 目标

本次工作目标是完成一次面向最终态的收敛清理，使仓库在以下几个方面一致：

1. 修复语法错误、启动级错误和明显坏掉的入口分支。
2. 移除高置信度异常无用功能，但不误删当前主路径或仍被承诺的能力。
3. 修正文档与实现不一致的问题，统一 README、帮助输出、CLI 入口和 Docker 文档说法。
4. 用现有测试与必要的 Docker 回归证明最终态成立。

## 非目标

以下内容不属于本次工作的范围：

- 不做与最终态收敛无关的大规模重构。
- 不为了“代码更优雅”而改变公开接口或行为。
- 不在飞书展示层通过补丁方式掩盖服务端/入口链路问题。
- 不主动删除“只是看起来可疑，但暂时无法证明应删”的功能。

## 推荐方案

采用“最终态主路径收敛”方案：

- 以当前仓库已经显式承诺的主路径能力为中心收敛实现与文档。
- 通过代码入口、公开文档、测试基线和近期提交四类事实来源交叉确认哪些能力属于最终态主路径。
- 对于删除动作，只有当某功能同时满足“当前主文档未承诺、help/入口未鼓励使用、无测试覆盖、与近期提交方向相悖、删除后不破坏主链路”中的大部分条件时，才作为删除候选。

相比“文档即真相”或“代码即真相”的方案，该方案更适合当前分支状态：既避免把历史残留实现合法化，也避免因文档暂未完全追上代码而误删仍需保留的能力。

## 范围边界

### 1. 语法与启动级错误

需要扫描并修复：

- shell 语法错误
- Python 导入或基础运行错误
- help、入口脚本、startup 等一执行就失败的错误

### 2. 异常无用功能

仅处理高置信度异常分支，例如：

- 已无调用入口、无文档承诺、无测试覆盖且与当前方向冲突的功能
- 明显是历史残留、半成品或坏路径的命令分支

### 3. 文档与实现不一致

需要收敛：

- README、CLI 参考、remote 文档、Docker 文档、help 输出之间的冲突
- 文档承诺存在但实现缺失/命名不符/参数不符的情况
- 实现存在但文档错误描述了行为的情况

### 4. 最终态一致性

需要统一：

- 公开命令面
- 推荐测试入口
- remote / token / regenerate-token / lark / Docker 相关主路径说明

## 明确保留项

结合当前仓库状态与已知方向，以下能力默认不作为清理目标：

- `remote-claude` 主入口
- `start` / `attach` / `list` / `status` / `kill`
- remote 相关能力：`connect`、`remote`、`token`、`regenerate-token`
- lark 主链路
- Docker 测试主链路
- 快捷启动器 `cla` / `cl` / `cx` / `cdx`（除非能证明已经损坏且应下线）

## 执行策略

### 阶段 1：建立事实基线

从四类来源建立当前最终态主路径：

- 代码入口：`bin/remote-claude`、`remote_claude.py`、相关 shell 脚本
- 公开文档：`README.md`、CLI/remote/docker 文档
- 测试基线：CLI、custom commands、startup、entry lazy init、Docker 回归相关测试
- 近期变更方向：最近提交与当前分支已修改内容

产出是一份内部判断：哪些是主路径、哪些是候选异常分支。

### 阶段 2：问题分桶

扫描结果分成三桶：

- A 桶：确定性错误（语法错误、help 错误、命令分发错误、文档明显写反）
- B 桶：一致性问题（文档与实现的名称、参数、示例不一致）
- C 桶：删除候选（无入口、无测试、无文档承诺、与最终态相悖的旧功能）

只有 C 桶涉及删除，且优先级低于 A/B 桶。

### 阶段 3：按主链路收敛

修改顺序固定为：

1. 先修入口与命令面
2. 再修文档
3. 最后删异常分支/无用功能

这样可以先稳定真实行为，再让文档对齐，最后再做瘦身。

### 阶段 4：验证闭环

验证分三层：

- 静态层：shell / Python 基本语法与可读性检查
- 单测层：优先运行 CLI、startup、custom commands、entry lazy init 相关测试
- Docker 层：若改动涉及 npm 打包、安装脚本、入口脚本或 Docker 逻辑，则补跑 Docker 回归

## 具体落点

### 命令入口与公开命令面

优先检查：

- `bin/remote-claude`
- `remote_claude.py`
- `scripts/_common.sh`
- `scripts/_help.sh`

重点验证：

- help 短路与真实命令分发是否一致
- README/CLI 文档里出现的命令是否都能从入口进入
- 是否存在文档未写、help 不暴露、但入口仍残留的异常命令分支
- `start`、`remote`、`token`、`regenerate-token`、`lark`、`uninstall` 等分支是否存在历史兼容残留或坏路径

### 文档主面

优先检查：

- `README.md`
- `docs/docker-test.md`
- CLI/remote 相关文档

重点验证：

- 推荐命令是否统一
- 示例是否与真实参数面一致
- 最终推荐路径是否只有一套说法
- Docker 文档是否与当前脚本真实行为一致

### Docker 与安装/启动链路

优先检查：

- `docker/scripts/docker-test.sh`
- npm / 安装相关脚本
- 入口包装脚本

重点验证：

- 文档宣称的 Docker 测试终态是否已由脚本真实实现
- 是否保留与当前文档脱节的旧测试/退出/诊断分支
- shell 包装是否存在仅为旧路径保留的异常逻辑

## 验收标准

### 功能验收

需要满足：

- `remote-claude` 公开命令面与 README/相关文档一致
- help 输出、shell 入口、Python 实现之间无明显冲突
- remote / token / regenerate-token / lark 等明确保留能力仍可达
- 删除的功能都能说明“为什么删、删后不影响主路径”

### 质量验收

需要满足：

- 不引入新的 shell/Python 语法错误
- 不在 Lark 展示层做掩盖式修补
- 不增加无关抽象或顺手重构
- 改动集中服务于修错、删坏路径和统一最终态说明

### 测试验收

若涉及启动链路、help、remote 参数或 shell 包装脚本，优先回归：

- `tests/test_custom_commands.py`
- `tests/test_cli_help_and_remote.py`
- `tests/test_startup_trace_logging.py`
- `tests/test_entry_lazy_init.py`

若涉及 npm 打包、安装脚本、入口脚本或 Docker 逻辑，补跑：

- `docker-compose -f docker/docker-compose.test.yml run --rm npm-test /project/docker/scripts/docker-test.sh`

必要时补充受影响模块的定向测试，但不无上限扩散。

### 文档验收

需要满足：

- README 作为用户主入口说明，不与详细文档冲突
- Docker 文档与当前脚本行为一致
- 术语、命令名、参数名、推荐用法统一
- 不再保留已经失效的旧路径说明

## 交付结果

完成后，应能明确回答：

1. 当前公开支持哪些命令与能力。
2. 删除了哪些内容，以及为什么可以删除。
3. 通过哪些测试证明最终态成立。
